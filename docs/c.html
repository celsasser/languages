The C Language

Published

March 28, 2026

Overview

The C Language

C is a general-purpose, procedural computer programming language supporting structured programming, lexical variable scope, and recursion. It provides constructs that map efficiently to typical machine instructions, making it the “lingua franca” of embedded systems and Digital Signal Processing (DSP). It sits at the intersection of high-level readability and low-level hardware control.

The 32 Keywords

The original ANSI C standard defined 32 keywords. These are reserved words that cannot be used as variable names. They form the skeleton of the language:

Data Types Control Flow Storage Classes Other
char if, else auto const
int switch, case, default extern volatile
short for, do, while register void
long break, continue static sizeof
float goto typedef enum
double return struct
signed union
unsigned

The Standard Library and I/O

C provides a rich Standard Library (often referred to as libc). Unlike higher-level languages, C does not have built-in statements for Input/Output (I/O). Instead, I/O is handled by functions in the standard library, primarily defined in <stdio.h>.

  • printf / scanf: The workhorses for formatted output and input.
  • fprintf / fscanf: Versions that work with file streams.
NoteWhat is stdout on Embedded Sytems?

In an embedded context (like the dsPIC): There is no “screen” or “keyboard” by default. The standard library functions must be “retargeted.” This usually means the low-level functions that printf calls (often named write or _mon_putc) are implemented by the developer to send characters byte-by-byte over a peripheral like UART (Universal Asynchronous Receiver-Transmitter). Thus, printf("Hello") effectively sends 5 bytes out of the serial port to your computer’s terminal.

The Compiler

A compiler is a program that translates computer code written in one programming language (the source language, like C) into another language (the target language, usually machine code or assembly). For the dsPIC, we typically use the XC16 compiler.

The Linker

A linker is a utility program that takes one or more object files generated by a compiler and combines them into a single executable file, library file, or another ‘object’ file. It resolves references between different modules (e.g., if main.c calls a function defined in dsp.c).

Stages of Compilation

The build process involves four distinct stages:

  1. Preprocessing: The preprocessor obeys commands that begin with # (like #include and #define). It expands macros and includes header files into the source.
  2. Compilation: The compiler translates the preprocessed source code into assembly language specific to the target architecture (e.g., dsPIC assembly).
  3. Assembly: The assembler translates the assembly code into machine code (binary instructions), producing an object file (.o).
  4. Linking: The linker combines the object files and necessary libraries to produce the final executable (e.g., .elf or .hex file) suitable for loading onto the hardware.

Sometimes stages 2 and 3 are combined and just called “compilation”, which is why you’ll see “three stages” in some references:

  1. Preprocessing
  2. Compilation (includes assembly)
  3. Linking

Preprocessing

The preprocessor provides directives that instruct the compiler to perform specific actions before the actual compilation begins.

  • #include: Inserts the contents of a specified file into the source code at the point where the directive appears. This is commonly used to include header files.
  • #define: Creates a macro, which is a fragment of code which has been given a name. Whenever the name is used, it is replaced by the contents of the macro.
  • #undef: Removes a macro definition.
  • #pragma: Provides the compiler with special instructions (e.g., configuring optimization levels or hardware-specific settings like configuration bits on the dsPIC). For example:
    • #pragma config FOSC = INTOSC sets the oscillator configuration.
    • #pragma GCC optimize ("O2") sets the optimization level.
  • #error: Aborts the compilation process and displays a specified error message. This is often used to ensure correct configuration options are selected.
  • __FILE__: A predefined macro that expands to the name of the current input file as a string literal. It is useful for logging and debugging.

Advanced Preprocessor Techniques

The preprocessor is more powerful than simple text substitution. Master these idioms for cleaner embedded code:

Conditional Compilation

Select code paths at compile time based on configuration:

#define DEBUG_LEVEL 2

#if DEBUG_LEVEL >= 2
    #define DEBUG_PRINT(fmt, ...) printf(fmt, ##__VA_ARGS__)
#elif DEBUG_LEVEL == 1
    #define DEBUG_PRINT(fmt, ...) printf(fmt "\n", ##__VA_ARGS__)
#else
    #define DEBUG_PRINT(fmt, ...) ((void)0)  // Compiles to nothing
#endif

// Usage
DEBUG_PRINT("Sensor value: %d", adc_result);

In production builds, change DEBUG_LEVEL to 0, and all debug prints vanish: zero code size, zero performance impact.

Stringification and Token Pasting

Stringification (#) converts a macro argument to a string:

#define STRINGIFY(x) #x
#define REGISTER_NAME(reg) printf("Register " #reg " = 0x%04X\n", reg)

REGISTER_NAME(AD1CON1);
// Expands to: printf("Register " "AD1CON1" " = 0x%04X\n", AD1CON1);

Token pasting (##) concatenates tokens:

#define GPIO_SET(port, pin) LAT##port##bits.LAT##port##pin = 1

GPIO_SET(B, 10);
// Expands to: LATBbits.LATB10 = 1

Practical use: Creating similar macros for all GPIO pins without copy-paste:

#define DEFINE_LED(name, port, pin) \
    static inline void name##_ON(void)  { LAT##port##bits.LAT##port##pin = 1; } \
    static inline void name##_OFF(void) { LAT##port##bits.LAT##port##pin = 0; } \
    static inline void name##_TOGGLE(void) { LAT##port##bits.LAT##port##pin ^= 1; }

// Define all three LED functions at once
DEFINE_LED(LED1, B, 10)
DEFINE_LED(LED2, B, 11)
DEFINE_LED(LED3, B, 12)

// Use
LED1_ON();
LED2_TOGGLE();

Variadic Macros

Macros that accept variable arguments using ... and __VA_ARGS__:

#define LOG_ERROR(fmt, ...) \
    do { \
        fprintf(stderr, "[ERROR] %s:%d: " fmt "\n", __FILE__, __LINE__, ##__VA_ARGS__); \
    } while(0)

// Usage
LOG_ERROR("Buffer overflow: size=%d, max=%d", buffer_size, MAX_SIZE);
// Expands to: fprintf(stderr, "[ERROR] audio.c:123: Buffer overflow: size=1024, max=512\n");

The do { ... } while(0) wrapper ensures the macro behaves like a single statement even with semicolon:

if(error_condition)
    LOG_ERROR("Something failed");  // Semicolon after macro
else
    continue_processing();

Without the wrapper, the else would attach incorrectly after macro expansion.

Include Guards vs. #pragma once

Prevent multiple inclusion:

Traditional

portable

// myheader.h
#ifndef MYHEADER_H
#define MYHEADER_H

// Header contents

#endif  // MYHEADER_H
Modern

non-standard but widely supported

// myheader.h
#pragma once

// Header contents

#pragma once is simpler and avoids name collisions, but include guards are guaranteed by the C standard.

Compilation

Compiler Attributes

__attribute__((..))

Important distinction: __attribute__ is NOT a preprocessor directive. It is a compiler extension (GCC and compatible compilers like XC16) that provides instructions to the compiler at compile time, not at preprocessing time.

Declaring Attributes of Functions - GCC Documentation:

…you can use function attributes to specify certain function properties that may help the compiler optimize calls or check code more carefully for correctness. For example, you can use attributes to specify that a function never returns (noreturn), returns a value depending only on the values of its arguments (const), or has printf-style arguments (format).

You can also use attributes to control memory placement, code generation options or call/return conventions within the function being annotated. Many of these attributes are target-specific. For example, many targets support attributes for defining interrupt handler functions, which typically must follow special register usage and return conventions. Such attributes are described in the subsection for each target. However, a considerable number of attributes are supported by most, if not all targets. Those are described in the Common Function Attributes section.

Syntax

The double parentheses are intentional and distinguish attributes from function calls:

__attribute__((attribute-list))
Common Attributes

For dsPIC Embedded Systems

Attribute Purpose Example
space(dma) Place variable in DMA-accessible RAM __attribute__((space(dma))) int16_t buf[128];
interrupt Function is an ISR (generates proper prologue/epilogue) __attribute__((interrupt)) void _ADC1Interrupt(void);
aligned(N) Align memory to N-byte boundary (required for DMA) __attribute__((aligned(128))) int16_t buffer[64];
section(".name") Place code/data in specific memory section __attribute__((section(".boot"))) void boot(void);
noreturn Function never returns (optimization hint) __attribute__((noreturn)) void fatal_error(void);
Memory placement example
// Place buffer in DMA-accessible RAM at a 128-byte aligned address
__attribute__((space(dma), aligned(128)))
int16_t audio_buffer[64];

// Without the attribute, the compiler places this in regular RAM
int16_t regular_buffer[64];  // May not be accessible by DMA!
Interrupt handler example
// Tells compiler to generate interrupt-safe entry/exit code
// - Use special interrupt return instruction (RETFIE)
// - Only save necessary registers (not all of them)
// - Disable interrupts during critical sections
void __attribute__((interrupt, auto_psv)) _T1Interrupt(void) {
    // ISR code here
    IFS0bits.T1IF = 0;  // Clear interrupt flag
}
Macros vs. Attributes

Key Difference

// ===== MACRO (Preprocessor) =====
#define DMA_BUF __attribute__((space(dma)))
DMA_BUF int16_t buffer[128];

// After PREPROCESSOR (text replacement occurs):
__attribute__((space(dma))) int16_t buffer[128];

// ===== ATTRIBUTE (Compiler) =====
__attribute__((space(dma))) int16_t buffer[128];
// Preprocessor IGNORES this line
// COMPILER reads: "Place this buffer in DMA memory region"
Why this Matters for DSP

  • space(dma): DMA controller can only access specific RAM regions. Without this attribute, your buffer might be placed where DMA cannot reach, causing silent failures.

  • aligned(N): Many DSP operations and DMA transfers require data aligned to specific boundaries. Misaligned data causes performance penalties or hardware faults.

  • interrupt: Generates optimized entry/exit code for ISRs, saving precious cycles; critical for real-time signal processing.

NoteMnemonic: No Hash, Not Preprocessor
  • Preprocessor directives start with #: #define, #include, #pragma
  • __attribute__ has no #, so it’s NOT a preprocessor feature
  • It’s a note from you to the compiler: “treat this declaration specially”

The C Language

C is characterized by its simplicity and efficiency. It uses a static type system to prevent unintended operations and manage memory usage. Key concepts include:

  • Pointers: Variables that store memory addresses, allowing direct memory manipulation.
  • Memory Management: Manual control over allocation (stack vs. heap), though in embedded DSP, static allocation is preferred to avoid non-deterministic behavior.
  • Bitwise Operations: Crucial for setting hardware registers (AND, OR, XOR, SHIFT).

Pointers

A pointer is a variable that stores the memory address of another variable. They are fundamental to C, allowing for:

  • Dynamic Memory Allocation: Allocating memory at runtime.
  • Array Manipulation: Accessing and iterating through array elements efficiently.
  • Pass-by-Reference: Modifying variable values inside functions.
  • Complex Data Structures: creating linked lists, trees, etc.

A pointer to a pointer is simply a pointer that stores the address of another pointer. This allows for multiple levels of indirection.

To declare a pointer, use the * operator: int *ptr; declares a pointer to an integer. To declare a pointer to a pointer, use two * operators: int **pptr;. To dereference a pointer (access the value it points to), use the * operator again.

#include <stdio.h>

int main() {
    int var = 10;
    int *ptr = &var;    // ptr holds the address of var
    int **pptr = &ptr;  // pptr holds the address of ptr

    printf("Value of var: %d\n", var);
    printf("Value via ptr: %d\n", *ptr);
    printf("Value via pptr: %d\n", **pptr);

    return 0;
}

Arrays and Pointers

The Duality

One of C’s most elegant (and confusing) features is the relationship between arrays and pointers. Understanding this is essential for writing efficient DSP code.

The Fundamental Truth

In most contexts, an array name decays to a pointer to its first element:

int buffer[256];
int *ptr = buffer;  // Equivalent to: int *ptr = &buffer[0];

Array subscripting is actually pointer arithmetic:

buffer[i]  ≡  *(buffer + i)  ≡  *(ptr + i)  ≡  ptr[i]

All four expressions compile to identical machine code.

Where They Differ

Arrays and pointers are not the same:

int arr[10];
int *ptr = arr;

sizeof(arr);  // Returns 40 (10 elements × 4 bytes each, on 32-bit)
sizeof(ptr);  // Returns 4 (size of the pointer itself)

&arr;  // Type: int (*)[10]  (pointer to array of 10 ints)
&ptr;  // Type: int**          (pointer to pointer to int)

Key distinction: An array is a fixed-size block of memory. A pointer is a variable holding an address.

Pointer Arithmetic for DSP

This equivalence makes DSP code elegant:

void process_samples(int16_t *input, int16_t *output, int length) {
    for(int i = 0; i < length; i++) {
        *output++ = (*input++) >> 1;  // Halve amplitude
    }
}

This is often faster than array indexing because:

  • No index register needed
  • Increment happens in parallel with memory access

Structures

Organizing Complex Data

Embedded DSP often requires managing related data. Structures group heterogeneous data types.

typedef struct {
    int16_t *buffer;      // Pointer to sample buffer
    uint16_t size;        // Buffer size
    uint16_t head;        // Write position
    uint16_t tail;        // Read position
    uint8_t  overrun;     // Error flag
} CircularBuffer_t;

// Initialize
CircularBuffer_t audio_in = {
    .buffer = input_samples,
    .size = 512,
    .head = 0,
    .tail = 0,
    .overrun = 0
};

// Use
audio_in.buffer[audio_in.head++] = new_sample;
if(audio_in.head >= audio_in.size) audio_in.head = 0;  // Wrap

Why use typedef struct? In C (unlike C++), without typedef you’d need to write struct CircularBuffer everywhere. With typedef, just CircularBuffer_t suffices.

Unions

Multiple Views of the Same Memory

A union allows storing different types in the same memory location. Only one member is valid at a time.

Example: Treating 32-bit data as bytes:

typedef union {
    uint32_t word;
    struct {
        uint8_t byte0;
        uint8_t byte1;
        uint8_t byte2;
        uint8_t byte3;
    } bytes;
} Word32_t;

Word32_t data;
data.word = 0x12345678;

// Access individual bytes without bit shifting
uint8_t high_byte = data.bytes.byte3;  // 0x12

DSP Use Case: Fast type punning (reinterpreting bits without conversion):

typedef union {
    float    f;
    int32_t  i;
    uint32_t u;
} FloatBits_t;

// Inspect binary representation of a float
FloatBits_t x = {.f = 3.14159f};
printf("Float bits: 0x%08X\n", x.u);
WarningAlignment Matters

Structures and unions must be properly aligned for efficient access. The dsPIC requires 16-bit values to be at even addresses. Use __attribute__((packed)) cautiously,it disables alignment, slowing access.

The const and static Keywords

Despite often appearing together in embedded code, const and static address entirely different concerns. const is a type qualifier: it constrains mutability, telling the compiler that a value must not be changed after initialization. static is a storage-class specifier: it governs lifetime and linkage, either extending a local variable’s lifetime to the duration of the program or restricting a global variable or function’s visibility to its translation unit. In short, const is about what can be done with data, while static is about where data lives and who can see it.

const

Putting Data in Flash

NoteConst Correctness

Const correctness is a programming principle of using the const keyword to explicitly enforce the immutability of data that is not intended to be modified. This practice helps prevent accidental data modifications, enhances code readability, and allows the compiler to catch potential errors at compile time, leading to more robust and maintainable code.

In embedded systems, RAM is precious, ROM/Flash is abundant. Use const to store read-only data in program memory:

// Without const: lives in RAM (scarce!)
int16_t filter_coeffs[128] = {123, 456, 789, ...};

// With const: lives in Flash (abundant!)
const int16_t filter_coeffs[128] = {123, 456, 789, ...};

On the dsPIC, accessing const data requires enabling PSV (Program Space Visibility). The compiler handles this automatically in most cases.

Const correctness also prevents accidental modification:

void apply_filter(const int16_t *input, int16_t *output, const int16_t *coeffs, int length);
//                ^^^^^^                                  ^^^^^^
//                Promises not to modify input or coefficients

static

Three Distinct Meanings

The static keyword is overloaded in C, with different meanings depending on context:

1. Static Local Variables

Function Scope

Retains value between function calls:

void count_calls(void) {
    static int count = 0;  // Initialized ONCE at program start
    count++;
    printf("Called %d times\n", count);
}

Use case: Persistent state without global pollution.

2. Static Global Variables

File Scope

Limits visibility to the current file:

// In audio.c
static int16_t internal_buffer[256];  // Only visible in audio.c

void process_audio(void) {
    // Can use internal_buffer here
}

// In main.c
// internal_buffer is NOT accessible here

Use case: Encapsulation, preventing name collisions across files.

3. Static Functions

File Scope

Function only visible within the file:

static void helper_function(void) {
    // Internal helper, not part of public API
}

The compiler can inline or optimize these more aggressively since it knows they’re never called externally.

Function Pointers

Callbacks and Dispatch Tables

Function pointers store the address of a function, enabling callbacks and runtime dispatch.

// Define a function pointer type
typedef void (*ProcessFunc_t)(int16_t *buffer, int length);

// Functions matching the signature
void echo_effect(int16_t *buf, int len) { /* ... */ }
void reverb_effect(int16_t *buf, int len) { /* ... */ }
void distortion_effect(int16_t *buf, int len) { /* ... */ }

// Use a function pointer
ProcessFunc_t current_effect = echo_effect;

void process_audio_block(int16_t *buffer, int length) {
    // Call through function pointer
    current_effect(buffer, length);
}

// Switch effects at runtime
void set_effect(int effect_number) {
    switch(effect_number) {
        case 0: current_effect = echo_effect; break;
        case 1: current_effect = reverb_effect; break;
        case 2: current_effect = distortion_effect; break;
    }
}

Dispatch table for efficient switching:

ProcessFunc_t effect_table[] = {
    echo_effect,
    reverb_effect,
    distortion_effect
};

#define NUM_EFFECTS (sizeof(effect_table) / sizeof(effect_table[0]))

void set_effect(int n) {
    if(n >= 0 && n < NUM_EFFECTS) {
        current_effect = effect_table[n];
    }
}

Memory Management

Memory management refers to the process of controlling and coordinating computer memory, assigning blocks to various running programs to optimize overall system performance.

Standard Functions

defined in <stdlib.h>

  • malloc(size_t size): Allocates a block of uninitialized memory.
  • calloc(size_t n, size_t size): Allocates memory for an array and initializes it to zero.
  • realloc(void *ptr, size_t size): Resizes a previously allocated memory block.
  • free(void *ptr): Deallocates memory, making it available for future use.
Where Does Memory Actually Live?

C has three distinct memory regions. Understanding the difference is essential for embedded work:

Region What Lives There Lifetime Size Known At
Stack Local variables, function arguments, return addresses Scope of the function Runtime (bounded by stack size)
Heap malloc/free allocations Until explicitly freed Runtime (unbounded)
Static / BSS Global variables, static variables, const data Entire program lifetime Compile time
NoteFixed Arrays: Stack or Static?

This is a common point of confusion. Fixed arrays declared at file/global scope or with the static keyword live in the BSS/data segment (static memory), not the stack. They’re allocated at compile time and exist for the entire duration of the program.

// --- STATIC MEMORY (BSS segment) ---
int16_t audio_buffer[512];          // Global: compile-time, lives forever
static int16_t filter_state[8];     // Static local: same region, but file-scoped

void process(void) {
    // --- STACK ---
    int16_t local_sample;           // Stack: only exists inside process()
    int16_t local_buf[16];          // Stack: gone when function returns!
}

When embedded programmers say “everything is in fixed arrays,” they mean global or static arrays — all occupying statically-allocated memory, known at compile time.

Strategies
  • Static Allocation: Memory is allocated at compile time and persists for the life of the program. This is the dominant strategy in safety-critical embedded systems: zero runtime overhead, completely deterministic, no fragmentation possible.
  • Dynamic Allocation: Memory is requested from the heap at runtime via malloc/free. Flexible, but carries risks of fragmentation, non-deterministic timing, and leaks.
  • Pool Allocation: A hybrid. A large static block is pre-allocated, then subdivided and managed at runtime as fixed-size chunks. You get dynamic-like flexibility with the determinism of static allocation.
  • No Allocation (Static-Only): Many embedded coding standards (MISRA-C, DO-178C for aviation, IEC 61508 for industrial safety) ban malloc/free entirely. All memory is declared at compile time. The tradeoff: you must know your worst-case memory needs up front.
Examples
Static Allocation
// Buffer lives in BSS segment for the entire program
static int16_t audio_buffer[512];
static int16_t filter_coeffs[64];

void process_audio(void) {
    for(int i = 0; i < 512; i++) {
        audio_buffer[i] = apply_filter(audio_buffer[i], filter_coeffs);
    }
}

Pros: Zero runtime cost, completely predictable, no fragmentation, no failure mode. Cons: Memory is always reserved even when not in use. Total memory usage must be known at design time.

Dynamic Allocation
void process_variable_length(int length) {
    int16_t *buffer = malloc(length * sizeof(int16_t));
    if(buffer == NULL) { handle_error(); return; }

    // Use buffer...

    free(buffer);   // Must not forget this, or memory leaks
}

Pros: Flexible sizes, only uses memory when needed. Cons: malloc can fail unpredictably, timing is non-deterministic, heap fragments over time.

Pool Allocation
#define POOL_BLOCK_SIZE  64   // Each block holds 64 samples
#define POOL_NUM_BLOCKS   8   // Pool has exactly 8 blocks

typedef struct {
    int16_t data[POOL_BLOCK_SIZE];
    uint8_t in_use;
} PoolBlock_t;

// The entire pool is statically allocated — this is the key
static PoolBlock_t memory_pool[POOL_NUM_BLOCKS];

PoolBlock_t *pool_alloc(void) {
    for(int i = 0; i < POOL_NUM_BLOCKS; i++) {
        if(!memory_pool[i].in_use) {
            memory_pool[i].in_use = 1;
            return &memory_pool[i];
        }
    }
    return NULL;  // Pool exhausted — deterministic failure, not a random crash
}

void pool_free(PoolBlock_t *block) {
    block->in_use = 0;
}

// Usage
PoolBlock_t *buf = pool_alloc();
if(buf) {
    fill_samples(buf->data);
    process(buf->data, POOL_BLOCK_SIZE);
    pool_free(buf);
}
  • Pros: No fragmentation (all blocks are the same size), bounded and deterministic (you know exactly how many allocations are possible at any time), fast, no heap required, failure mode is explicit (NULL return) rather than silent.
  • Cons: Wastes memory when block sizes don’t match your needs. Cannot handle variable-size requests without maintaining multiple pools of different sizes.

The Build

The “build” is the process of converting your entire project (source files, headers, configuration

bits) into a machine-readable format.

In an embedded context, this output is often a HEX file. This file is not run by an OS like an .exe on Windows; instead, it is “flashed” directly into the microcontroller’s non-volatile program memory.

The build process also generates a Map File, which is critical for debugging. It shows exactly where every function and variable is placed in the device’s memory.

Make

Make is a build automation tool that automatically builds executable programs and libraries from source code by reading files called Makefiles.

It is essential because it tracks dependencies. If you modify one .c file in a project with 50 files, Make knows to only recompile that specific file and then re-link, rather than rebuilding everything from scratch.

A simple rule in a Makefile looks like this:

target: prerequisites
    recipe
  • Target: The file to be generated (e.g., main.o).
  • Prerequisites: The files the target depends on (e.g., main.c, main.h).
  • Recipe: The command to create the target (e.g., xc16-gcc -c main.c ...).

Example

A Text Processor

Here is a complete example of a C program that reads text from standard input, converts it to uppercase, and writes it to standard output. We also provide the Makefile to build it.

The Source Code

upper.c

#include <stdio.h>
#include <ctype.h>

int main(void) {
    int c;

    // Read until End-Of-File (EOF)
    // getchar() returns the next character or EOF
    while ((c = getchar()) != EOF) {
        // Transform the character
        int upper = toupper(c);

        // Write the character back to stdout
        putchar(upper);
    }

    return 0;
}

Optimization

Optimization allows the compiler to adjust the code to improve performance or reduce size, often at the expense of compilation time and debuggability.

Compiler Optimization Levels

  • -O0: No optimization. Best for debugging as code flow matches source lines exactly.
  • -O1: Moderate optimization.
  • -O2: High optimization for speed.
  • -Os: Optimize for size (crucial when flash memory is limited).

The volatile Keyword

In embedded C, volatile is perhaps the most important qualifier. It tells the compiler: “The value of this variable may change at any time, without any action being taken by the code the compiler finds nearby.”

This prevents the compiler from optimizing away reads/writes to:

  • Hardware Registers: A status flag might change due to an interrupt, not software flow.
  • Global Variables: Shared between an Interrupt Service Routine (ISR) and the main loop.
// Without volatile, the compiler might replace this loop with a generic 'while(true)'
// or remove the read entirely if it thinks the value never changes.
volatile int *status_reg = 0xFF00;
while (*status_reg == 0); // Wait for hardware flag

Features and Libraries for DSP

Digital Signal Processing requires specific features to handle continuous signals efficiently.

Fixed-Point Arithmetic

While floating-point is easy to use, many DSCs (Digital Signal Controllers) are optimized for fixed-point arithmetic. This represents fractional numbers using integers, trading dynamic range for speed and determinism.

Fractional Representation

1.15 Format

NoteQ15

Q15 (or 1.15) is a fixed-point number format where:

  • 1 bit is used for the sign.
  • 15 bits are used for the fractional part. This allows representing numbers in the range \([-1, 0.999969]\). It is widely used in 16-bit DSPs because it maps perfectly to the 16-bit integer registers.

In 16-bit DSPs, we often use the Q15 (or 1.15) format.

  • The 16 bits are treated as a signed integer range \([-32768, +32767]\).
  • Conceptually, we divide this integer by \(2^{15}\) (32768).
  • Range: \(-1.0\) to \(\approx 0.999969\).
  • 0x7FFF (32767) represents \(\approx 1.0\).
  • 0x8000 (-32768) represents \(-1.0\).

Arithmetic: When you multiply two Q15 numbers, the result is conceptually Q30 (since \(2^{-15} \times 2^{-15} = 2^{-30}\)). To store this back into a 16-bit Q15 variable, you must shift the result to the right by 15 bits.

\[ Result_{Q15} = (A_{Q15} \times B_{Q15}) >> 15 \]

DSP hardware often does this shift (and handles saturation so overflows don’t wrap around to negative numbers) automatically in a single cycle.

DSP Extensions and Intrinsics

Standard C does not natively support specialized DSP hardware instructions like MAC (Multiply-and-Accumulate) with saturation. To bridge this, compilers provide intrinsics , function-like commands that map directly to assembly instructions.

// Example of a built-in intrinsic for dsPIC (XC16)
// Performs a 16x16 multiply and accumulates to an accumulator
__builtin_mac(a, b, &accumulator, ...);

Circular Buffers

DSP algorithms (like FIR filters) often require sliding windows over data. Circular buffers allow the pointer to wrap around automatically when it reaches the end of the buffer, often supported by hardware modulo addressing.

The DSP Library

Microchip provides a DSP library (often included with the compiler) containing optimized functions for:

  • Vector operations (add, subtract, dot product)
  • Filters (FIR, IIR)
  • Transforms (FFT)

Using these pre-validated libraries is often more efficient than writing raw C implementations.

Common Pitfalls and Best Practices

C gives you enormous power and freedom, but with that comes the responsibility to avoid subtle bugs. Here are the most common traps that snare even experienced programmers.

1. Integer Promotion and Implicit Conversions

C’s type promotion rules are surprising. In expressions mixing different integer types, smaller types are promoted to larger types.

int16_t a = 30000;
int16_t b = 30000;

// Wrong! Overflow before assignment
int32_t result = a + b;  // a + b happens in 16-bit: -5536
                         // Then promoted to int32_t

// Correct: Promote before arithmetic
int32_t result = (int32_t)a + b;  // First term is int32_t, b is promoted

Rule of thumb: When doing arithmetic that might overflow, explicitly cast the first operand to the desired result type.

2. Operator Precedence Surprises

// Common mistake
if (flags & MASK == ENABLED)  // Wrong! Parsed as: flags & (MASK == ENABLED)

// Correct
if ((flags & MASK) == ENABLED)  // Mask first, then compare

Bitwise operators (&, |, ^) have lower precedence than comparison operators (==, !=). Always use parentheses.

3. Assignment vs. Comparison

if (x = 5)  // Oops! Assignment, not comparison
            // Always true (5 is non-zero)

if (x == 5) // Correct

Defensive programming: Some teams write constants first (Yoda conditions):

if (5 == x)  // If you accidentally type =, compiler error!

4. Uninitialized Variables

int16_t sample;  // Garbage value!
process_sample(sample);  // Undefined behavior

int16_t sample = 0;  // Always initialize

Global and static variables are automatically zero-initialized. Local variables are not.

5. Buffer Overruns

The classic C vulnerability:

char buffer[10];
strcpy(buffer, "This string is way too long");  // Buffer overflow!

// Safe alternatives
strncpy(buffer, source, sizeof(buffer) - 1);
buffer[sizeof(buffer) - 1] = '\0';  // Ensure null termination

Or better yet, use snprintf:

snprintf(buffer, sizeof(buffer), "Value: %d", x);

6. Off-by-One Errors

int array[10];
for(int i = 0; i <= 10; i++) {  // Wrong! Accesses array[10] which doesn't exist
    array[i] = 0;
}

// Correct
for(int i = 0; i < 10; i++) {
    array[i] = 0;
}

Mnemonic: Use < for counting up to a size, > for counting down to zero.

7. Pointer Arithmetic Gone Wrong

int16_t *ptr = malloc(10 * sizeof(int16_t));
ptr++;  // Now points to second element

free(ptr);  // WRONG! Must free the original pointer

Rule: Never free() a pointer that has been modified. Keep the original pointer.

8. Modifying String Literals

char *str = "Hello";  // String literal in read-only memory
str[0] = 'h';         // Undefined behavior! Might crash

// Correct: Array on stack (modifiable)
char str[] = "Hello";
str[0] = 'h';  // OK

9. Forgetting break in switch Statements

switch(state) {
    case STATE_IDLE:
        init_processing();
        // Oops! Falls through to STATE_ACTIVE
    case STATE_ACTIVE:
        process_data();
        break;
}

Fallthrough is rarely intentional. Always use break unless you explicitly want fallthrough (and comment it!):

case STATE_WARNING:
    flash_led();
    // FALLTHROUGH  (intentional)
case STATE_ERROR:
    halt_system();
    break;

10. Macro Side Effects

#define MAX(a,b) ((a) > (b) ? (a) : (b))

int x = 5;
int result = MAX(x++, 10);  // x incremented TWICE if x > 10!

Macros expand textually. If the argument has side effects, they happen multiple times.

Solution: Use inline functions:

static inline int max(int a, int b) {
    return (a > b) ? a : b;
}

11. Undefined Behavior

The Worst Kind of Bug

Some operations have undefined behavior: the C standard places no requirements on what happens. The program might work, crash, or corrupt data silently.

Common sources
  • Signed integer overflow: INT_MAX + 1 is undefined (use unsigned for modulo arithmetic)
  • Null pointer dereference: *ptr where ptr == NULL
  • Accessing freed memory: “use after free”
  • Array out of bounds: arr[100] when arr has size 50
  • Uninitialized variables: Reading before writing
  • Violating strict aliasing: Casting pointer types incorrectly
WarningStupid Optimizer

The optimizer assumes undefined behavior never happens and optimizes accordingly, potentially causing bizarre bugs in release builds that don’t appear in debug builds.

Interrupts

Interrupts are the mechanism by which hardware events can pause the normal flow of your program, execute a short handler function, and then resume where it left off. In embedded DSP systems, interrupts are essential, they allow your processor to respond to time-critical events (ADC samples ready, timer ticks, button presses) without constantly polling hardware.

What Are Interrupts?

Think of interrupts as hardware-triggered function calls. When an event occurs (e.g., a timer expires), the processor:

  1. Pauses the currently executing code (saves context)
  2. Jumps to a special function called an Interrupt Service Routine (ISR)
  3. Executes the ISR
  4. Returns to where it left off (restores context)

All of this happens automatically in hardware, you just write the ISR.

Why use interrupts instead of polling?
// Polling (wasteful)
while(1) {
    if(TIMER_FLAG) {
        do_something();
        TIMER_FLAG = 0;
    }
    // CPU sits here burning power, checking constantly
}

// Interrupt-driven (efficient)
void __attribute__((interrupt)) _T1Interrupt(void) {
    do_something();
    IFS0bits.T1IF = 0;  // Clear flag
}

// Main can do other work or sleep
while(1) {
    do_other_work();  // ISR interrupts this when needed
}

Anatomy of an ISR

dsPIC

Here’s a complete timer interrupt example for the dsPIC33F:

// Global counter incremented by ISR
volatile unsigned int timer1Count = 0;

// Timer 1 ISR - called every time Timer1 expires
void __attribute__((interrupt, auto_psv)) _T1Interrupt(void)
{
    timer1Count++;           // Increment counter
    IFS0bits.T1IF = 0;       // Clear interrupt flag (REQUIRED!)

    // Keep ISRs SHORT! No printf, no delays, no heavy math.
}
Key elements
  1. __attribute__((interrupt, auto_psv)): Tells compiler this is an ISR
    • Automatically saves/restores registers
    • auto_psv handles program space visibility for dsPIC
  2. ISR name: Must match the vector name (e.g., _T1Interrupt for Timer 1)
    • See device datasheet for all vector names
    • Common ones: _T1Interrupt, _T2Interrupt, _ADC1Interrupt, _INT0Interrupt
  3. Clear the interrupt flag: IFS0bits.T1IF = 0;
    • If you forget this, the ISR will immediately trigger again!
    • Each interrupt source has a flag bit
  4. No return type, no parameters: ISRs are called by hardware, not by your code

Communicating Between ISR and Main

ISRs and main code run asynchronously, the ISR can fire at any moment. To share data safely:

1. Use Global Variables

// In timer.c
volatile unsigned int timer1Count = 0;  // Modified by ISR

void __attribute__((interrupt, auto_psv)) _T1Interrupt(void) {
    timer1Count++;
    IFS0bits.T1IF = 0;
}
// In main.c
extern volatile unsigned int timer1Count;  // Access the same variable

int main(void) {
    timer1Init(15625);  // Start timer interrupts

    while(1) {
        if(timer1Count >= 10) {  // Check flag set by ISR
            timer1Count = 0;
            toggle_led();
        }
    }
}
Why volatile?

The volatile keyword tells the compiler: “This variable can change at any time (by an ISR), so always read it from memory, never cache it in a register.”

Without volatile, the compiler might optimize:

while(timer1Count < 10);  // Compiler sees: "timer1Count never changes in this loop"
                          // Optimizes to: while(1);  (infinite loop!)

With volatile:

volatile unsigned int timer1Count;
while(timer1Count < 10);  // Compiler: "Must check memory every time"

Rule of thumb: Any variable accessed by both ISR and main must be volatile.

2. Flag-Based Pattern

A cleaner pattern for event-driven code:

volatile uint8_t button_pressed = 0;  // Flag set by ISR

// Button press ISR
void __attribute__((interrupt, auto_psv)) _INT0Interrupt(void) {
    button_pressed = 1;  // Set flag
    IFS0bits.INT0IF = 0;
}

// Main loop
int main(void) {
    init_button_interrupt();

    while(1) {
        if(button_pressed) {
            button_pressed = 0;     // Clear flag
            handle_button_event();  // Do heavy processing here
        }

        // Do other work...
    }
}

Philosophy: ISR does minimal work (set a flag), main loop does the heavy lifting.

3. Atomic Operations

If you need to read/write multi-byte variables that have a larger bit width than the CPU’s word size (16 bits for dsPIC), you need protection for atomic access. One way is to disable interrupts temporarily:

volatile uint32_t sample_count = 0;  // Modified by ISR

// Reading in main (without protection)
uint32_t count = sample_count;  // DANGER! Could be interrupted mid-read
                                // (16-bit CPU reads 32-bit value in 2 operations)

// Safe version
uint32_t get_sample_count(void) {
   uint32_t count;
   uint16_t saved_ipl = SRbits.IPL;  // Save current interrupt priority
   SRbits.IPL = 7;                   // Disable all interrupts (priority 7)
   count = sample_count;             // Now atomic
   SRbits.IPL = saved_ipl;           // Restore interrupts
   return count;
}

Note: For single-byte or single-word (16-bit on dsPIC) variables, reads/writes are inherently atomic: no protection needed.

Common Interrupt Patterns

Timer-Based Real-Time Clock

// Create a millisecond timebase
volatile uint32_t millis = 0;

void __attribute__((interrupt, auto_psv)) _T1Interrupt(void) {
    millis++;  // Increments every 1ms
    IFS0bits.T1IF = 0;
}

// Use in main
void delay_ms(uint32_t ms) {
    uint32_t start = millis;
    while((millis - start) < ms);  // Wait
}

int main(void) {
    timer1_init_1ms();  // Configure for 1ms ticks

    while(1) {
        LED = 1;
        delay_ms(500);
        LED = 0;
        delay_ms(500);
    }
}

ADC Data Acquisition

#define BUFFER_SIZE 512
volatile int16_t adc_buffer[BUFFER_SIZE];
volatile uint16_t buffer_index = 0;
volatile uint8_t buffer_ready = 0;

void __attribute__((interrupt, auto_psv)) _ADC1Interrupt(void) {
    adc_buffer[buffer_index++] = ADC1BUF0;  // Read sample

    if(buffer_index >= BUFFER_SIZE) {
        buffer_index = 0;
        buffer_ready = 1;  // Signal main: buffer full
    }

    IFS0bits.AD1IF = 0;
}

int main(void) {
    adc_init();

    while(1) {
        if(buffer_ready) {
            buffer_ready = 0;
            process_buffer(adc_buffer, BUFFER_SIZE);  // FFT, filter, etc.
        }
    }
}

Common Pitfalls

1. Forgetting volatile

uint16_t flag = 0;  // WRONG! Compiler may optimize away

void ISR(void) { flag = 1; }

while(!flag);  // May loop forever if compiler caches flag in register

Fix: volatile uint16_t flag = 0;

2. ISRs That Are Too Long

void __attribute__((interrupt)) _T1Interrupt(void) {
    printf("Timer fired\n");  // VERY BAD! printf is slow
    delay_ms(100);            // TERRIBLE! Blocks other interrupts
    compute_fft();            // NO! Do this in main
    IFS0bits.T1IF = 0;
}

ISR golden rule: Get in, do minimal work, get out. Typically < 10-20 µs.

Good ISR: Set flags, read/write hardware registers, increment counters.

Bad ISR: Heavy computation, floating-point math, I/O operations, long delays.

3. Not Clearing Interrupt Flags

void __attribute__((interrupt)) _T1Interrupt(void) {
    do_something();
    // Forgot: IFS0bits.T1IF = 0;
}
// ISR will re-trigger immediately, hanging your system!

4. Re-entrancy Issues

An ISR can interrupt itself if you nest interrupts. Generally, keep default behavior (ISRs don’t interrupt each other) unless you have a good reason.

5. Stack Overflow

Each interrupt uses stack space. Too many nested interrupts or large local variables in ISRs can overflow the stack.

Best practice: Use global or static variables in ISRs, not large local arrays.

Setting Up Interrupts

dsPIC Example

void timer1_init(uint16_t period) {
    // Configure Timer1
    T1CON = 0x0000;              // Stop timer, clear settings
    TMR1 = 0;                    // Clear counter
    PR1 = period;                // Set period

    // Configure interrupt
    IPC0bits.T1IP = 4;           // Set priority (1-7, 0=disabled)
    IFS0bits.T1IF = 0;           // Clear flag
    IEC0bits.T1IE = 1;           // Enable interrupt

    // Start timer
    T1CONbits.TON = 1;
}
Key steps
  1. Configure the peripheral (timer, ADC, etc.)
  2. Set interrupt priority (IPCx registers)
  3. Clear interrupt flag (IFSx registers)
  4. Enable interrupt (IECx registers)

Quick Reference

Real-World Example

LED Blink with Timer

Putting it all together, the example from your code:

// timer.c
volatile unsigned int timer1Count = 0;

void __attribute__((interrupt, auto_psv)) _T1Interrupt(void) {
    timer1Count++;
    IFS0bits.T1IF = 0;
}

void timer1Init(uint16_t period) {
    T1CON = 0x0030;      // Prescaler 1:256
    TMR1 = 0;
    PR1 = period;
    IPC0bits.T1IP = 4;
    IFS0bits.T1IF = 0;
    IEC0bits.T1IE = 1;
    T1CONbits.TON = 1;
}
// main.c
#include "timer.h"

extern volatile unsigned int timer1Count;

int main(void) {
    init_hardware();

    // Timer interrupts every 100ms
    timer1Init(15625);  // (40MHz / 4) / 256 / 15625 = 10Hz

    while(1) {
        RED_LED = 1;
        while(timer1Count < 10);  // Wait 1 second (10 × 100ms)
        timer1Count = 0;

        RED_LED = 0;
        while(timer1Count < 10);
        timer1Count = 0;
    }
}
How it works
  • Timer 1 fires every 100ms, incrementing timer1Count in the ISR
  • Main loop waits for counter to reach 10 (1 second), then resets it
  • LED blinks on/off with 1 second periods

This is exactly the pattern you encountered; now you know what’s happening behind the scenes!

Embedded C Idioms

State Machines

Many embedded systems are best modeled as finite state machines:

typedef enum {
    STATE_IDLE,
    STATE_ACQUIRING,
    STATE_PROCESSING,
    STATE_ERROR
} State_t;

static State_t current_state = STATE_IDLE;

void state_machine_update(void) {
    switch(current_state) {
        case STATE_IDLE:
            if(start_button_pressed()) {
                current_state = STATE_ACQUIRING;
                init_acquisition();
            }
            break;

        case STATE_ACQUIRING:
            if(buffer_full()) {
                current_state = STATE_PROCESSING;
            } else if(timeout_occurred()) {
                current_state = STATE_ERROR;
            }
            break;

        case STATE_PROCESSING:
            process_buffer();
            current_state = STATE_IDLE;
            break;

        case STATE_ERROR:
            handle_error();
            current_state = STATE_IDLE;
            break;
    }
}
Benefits
  • Clear, maintainable code
  • Easy to visualize as a diagram
  • Deterministic behavior

Ring Buffers

Circular Buffers

Essential for continuous data streams (audio, serial data):

#define BUFFER_SIZE 256  // Must be power of 2 for fast modulo

typedef struct {
    int16_t buffer[BUFFER_SIZE];
    volatile uint16_t head;  // Write position (ISR writes)
    volatile uint16_t tail;  // Read position (main reads)
} RingBuffer_t;

// ISR writes
void buffer_write(RingBuffer_t *rb, int16_t value) {
    rb->buffer[rb->head] = value;
    rb->head = (rb->head + 1) & (BUFFER_SIZE - 1);  // Fast modulo
}

// Main loop reads
int16_t buffer_read(RingBuffer_t *rb) {
    int16_t value = rb->buffer[rb->tail];
    rb->tail = (rb->tail + 1) & (BUFFER_SIZE - 1);
    return value;
}

// Check if data available
int buffer_available(RingBuffer_t *rb) {
    return (rb->head - rb->tail) & (BUFFER_SIZE - 1);
}

Why power of 2? Modulo becomes a simple AND: x % 256 ≡ x & 255 (much faster).

Why volatile? The ISR and main loop access these independently; the compiler must not optimize away reads.

Bit Fields for Hardware Registers

typedef struct {
    unsigned pin0  : 1;
    unsigned pin1  : 1;
    unsigned pin2  : 1;
    unsigned       : 5;  // Reserved bits
    unsigned mode  : 2;
    unsigned       : 6;  // Reserved
} ControlRegister_t;

volatile ControlRegister_t *ctrl_reg = (ControlRegister_t*)0x0040;

// Access individual bits
ctrl_reg->pin0 = 1;
ctrl_reg->mode = 2;

Caveat: Bit field layout is compiler-dependent. For portable code, use bitwise operations instead.

TRISCBITS is a bit field and is used to control the direction of pins on Port C: whether each pin is an input or output. This struct overlays the 16-bit TRISC hardware register, giving you named access to individual bits instead of having to manually mask and shift.

It’s in p33FJ256GP506.h which is included via xc.h (which is the chip selector). We use it in BlinkBlocking.

#define TRISC TRISC
extern volatile uint16_t TRISC __attribute__((__sfr__));
typedef struct tagTRISCBITS {
  uint16_t :1;
  uint16_t TRISC1:1;
  uint16_t TRISC2:1;
  uint16_t :9;
  uint16_t TRISC12:1;
  uint16_t TRISC13:1;
  uint16_t TRISC14:1;
  uint16_t TRISC15:1;
} TRISCBITS;
extern volatile TRISCBITS TRISCbits __attribute__((__sfr__));

Breaking down TRISCBITS:

  • uint16_t :1; - Anonymous 1-bit field (bit 0) - reserved/unused pin
  • uint16_t TRISC1:1; - Named 1-bit field for pin C1 (bit 1)
  • uint16_t TRISC2:1; - Pin C2 (bit 2)
  • uint16_t :9; - Anonymous 9-bit field (bits 3-11) - pins that don’t exist on Port C
  • uint16_t TRISC12:1; - Pins C12 (bit 12)
  • uint16_t TRISC13:1; - Pins C13 (bit 13)
  • uint16_t TRISC14:1; - Pins C14 (bit 14)
  • uint16_t TRISC15:1; - Pins C15 (bit 15)

The TRISC register controls pin direction on Port C:

  • TRISCbits.TRISC1 = 0; → Pin C1 is an output
  • TRISCbits.TRISC1 = 1; → Pin C1 is an input
