Why Python Exists

Python was created by Guido van Rossum in the late 1980s and first released in 1991. Van Rossum was working at CWI in the Netherlands and wanted a language that filled the gap between shell scripting (too limited) and C (too verbose for rapid development). He named it after Monty Python, not the snake.

The motivation was philosophical as much as practical: productivity matters. A program you can write in an afternoon and read back six months later is often more valuable than a highly optimized one that takes weeks to get right. Python made a deliberate bet that developer time is usually the bottleneck, not CPU time.

That bet paid off. Python is now the dominant language for data science, machine learning, scientific computing, web backends, and automation. It runs everywhere from Raspberry Pis to GPU clusters.

The Core Design Decisions

1. Everything is an object.

Every value in Python — integers, strings, functions, classes, modules — is a first-class object with an identity, a type, and a value. There are no primitives in the C sense. This uniformity makes the language consistent and extensible, at the cost of memory overhead.

2. Dynamic typing.

Variables have no declared type. A variable is a name bound to an object, and that binding can be rebound to any object at any time. The type lives with the object, not the variable. This eliminates boilerplate but shifts type errors from compile time to runtime.

3. Reference semantics throughout.

Assignment never copies an object. It copies a reference to an object. This has profound implications for mutable data, covered in depth below.

4. Automatic memory management.

CPython (the reference implementation) uses reference counting as its primary garbage collection strategy, augmented by a cycle collector for circular references. Memory is managed for you, but understanding how it works prevents subtle bugs.

5. Readability as a design constraint.

Python’s syntax enforces structure through indentation rather than braces. The language includes exactly one obvious way to do most things, by design. This is not a limitation: it is a feature that makes code written by others readable.

6. Batteries included.

The standard library is enormous. Networking, file I/O, JSON, regular expressions, HTTP, CSV, database access, unit testing: most of what you need for a typical program is already there.

Modules

Every .py file is a module. Its name is the filename without the extension. You import it with import:

# math_utils.py
def add(a, b):
    return a + b

# main.py
import math_utils
print(math_utils.add(2, 3))

# or import specific names directly
from math_utils import add
print(add(2, 3))

Packages

A directory containing an __init__.py file is a package: a namespace that groups related modules.

my_project/
├── main.py
└── geometry/
    ├── __init__.py       # makes geometry/ a package
    ├── shapes.py         # geometry.shapes
    └── transforms.py     # geometry.transforms

__init__.py can be empty or can run initialization code and re-export names to shape the package’s public API — the Python equivalent of pub use in Rust.

Virtual Environments

Python is installed system-wide, but dependencies differ per project. A virtual environment is an isolated Python installation for a single project:

python -m venv .venv            # create a virtual environment
source .venv/bin/activate       # activate it (macOS/Linux)
pip install requests            # install into this environment only

Dependencies are recorded in pyproject.toml or requirements.txt. Always use a virtual environment; installing packages system-wide causes conflicts across projects.

The Basics

Variables and Types

Python variables are not declarations: they are name bindings. A name comes into existence the moment you assign to it, and its type is determined by the object it is bound to:

x = 42          # x is bound to an int object
x = "hello"     # x is now bound to a str object — no error
x = [1, 2, 3]   # x is now bound to a list object

The built-in scalar types:

i = 42              # int (arbitrary precision: no overflow ever)
f = 3.14            # float (64-bit IEEE 754)
c = 2 + 3j          # complex
b = True            # bool (subclass of int: True == 1, False == 0)
s = "hello"         # str (immutable Unicode sequence)
n = None            # NoneType: the absence of a value (Python's null)

No integer overflow. Python integers are arbitrary precision — they grow as large as memory allows. This is convenient and occasionally surprising in performance-sensitive code.

Type Hints

Advisory types, not enforced types

Type annotations exist but are not enforced at runtime. The interpreter ignores them entirely. They exist for static type checkers (mypy, pyright) and as documentation. This is the opposite of Rust, where types are mandatory and compiler-checked. In Python, types are advisory — useful but optional.

Basic Annotations

Variables and function signatures are annotated with : for values and parameters and -> for return types:

name: str = "alice"
count: int = 0

def add(a: int, b: int) -> int:
    return a + b

def greet(name: str) -> None:   # None means the function returns nothing
    print(f"hello, {name}")

Container Types

Built-in containers are generic as of Python 3.9. Use lowercase directly:

def total(numbers: list[int]) -> int:
    return sum(numbers)

def lookup(data: dict[str, int], key: str) -> int:
    return data[key]

coords: tuple[float, float] = (3.0, 4.0)
unique: set[str] = {"a", "b", "c"}

Before Python 3.9 the same types came from the typing module: List[int], Dict[str, int]. You will see both styles in existing code.

Optional Values

T | None marks a value that might be absent. It is Python’s loose equivalent of Option<T> in Rust, but the runtime does not enforce it:

def find_user(user_id: int) -> str | None:
    if user_id == 1:
        return "alice"
    return None

Older code uses Optional[str] from the typing module, which means exactly the same thing. The | syntax is available from Python 3.10 onward.

Union Types

A | B means a value that can be either type:

def process(value: int | str) -> str:
    return str(value)

Older code writes Union[int, str] from typing. Same meaning.

Any

Any is the escape hatch. A value typed as Any is compatible with every other type in both directions. Use it when interfacing with untyped code or when the type is genuinely dynamic:

from typing import Any

def log(value: Any) -> None:
    print(value)

Overusing Any defeats the purpose of annotations. Treat it like unsafe in Rust: reach for it when necessary, not by default.

Callable

Functions passed as arguments can be typed:

from typing import Callable

def apply(func: Callable[[int], int], value: int) -> int:
    return func(value)

apply(lambda x: x * 2, 5)   # 10

Callable[[ArgType, ...], ReturnType] — the list contains parameter types, the second argument is the return type.

Generic Functions with TypeVar

TypeVar defines a placeholder type resolved at the call site, the same concept as a generic type parameter in Rust:

from typing import TypeVar

T = TypeVar("T")

def first(items: list[T]) -> T:
    return items[0]

first([1, 2, 3])         # return type inferred as int
first(["a", "b", "c"])  # return type inferred as str

Without TypeVar you would annotate the return as Any and lose the inference. With it, the type checker knows the return type matches whatever the list contains.

Protocol

Structural Subtyping

Protocol is Python’s formal answer to duck typing. It defines an interface by the methods a type must have, without requiring explicit inheritance. This is the closest Python analog to Rust traits:

from typing import Protocol

class Drawable(Protocol):
    def draw(self) -> None: ...

class Circle:
    def draw(self) -> None:
        print("drawing circle")

class Square:
    def draw(self) -> None:
        print("drawing square")

def render(shape: Drawable) -> None:   # accepts anything with a draw() method
    shape.draw()

render(Circle())   # fine
render(Square())   # fine

Neither Circle nor Square inherits from Drawable. The type checker verifies structural compatibility at the call site. This is duck typing formalized: if it has the right methods, it qualifies.

Final and Literal

Final declares a constant — a value that should never be reassigned:

from typing import Final

MAX_RETRIES: Final = 3

Literal constrains a parameter to a specific set of values. The type checker rejects anything outside the set:

from typing import Literal

def set_direction(d: Literal["north", "south", "east", "west"]) -> None:
    ...

set_direction("up")   # type checker error
WarningAnnotations do not run at runtime

None of this affects execution. If you pass the wrong type at runtime, Python will not stop you — it will simply fail when an operation on the wrong type fails. Run mypy or configure pyright in your editor to get the errors before runtime.

Functions

def greet(name):
    print(f"Hello, {name}!")        # f-string: embed expressions in strings

def add(a, b):
    return a + b

# default arguments
def connect(host, port=8080, timeout=30):
    ...

# variable positional arguments
def total(*numbers):
    return sum(numbers)

total(1, 2, 3, 4)   # 10

# variable keyword arguments
def create_user(**fields):
    print(fields)   # {'name': 'alice', 'role': 'admin'}

create_user(name="alice", role="admin")

Functions are first-class objects. They can be stored in variables, passed as arguments, and returned from other functions.

Control Flow

# if / elif / else
if x > 0:
    print("positive")
elif x == 0:
    print("zero")
else:
    print("negative")

# for iterates over any iterable
for item in [1, 2, 3]:
    print(item)

for i in range(10):                     # 0 to 9
    print(i)

for i, item in enumerate(["a", "b", "c"]):
    print(i, item)                      # 0 a, 1 b, 2 c

# while
while x > 0:
    x -= 1

for in Python iterates over any iterable: lists, strings, dicts, files, generators, database cursors. There is no index-based loop by default; use enumerate when you need the index.

Comprehensions

Python’s idiomatic way to build collections

Comprehensions are a concise syntax for building a new collection by transforming or filtering an existing iterable. They are the Python equivalent of chaining .map(), .filter(), and .collect() in Rust, but written as a single inline expression.

List Comprehensions

The foundation. [expression for item in iterable]:

squares = [x ** 2 for x in range(10)]
# [0, 1, 4, 9, 16, 25, 36, 49, 64, 81]

Add an if clause to filter:

even_squares = [x ** 2 for x in range(10) if x % 2 == 0]
# [0, 4, 16, 36, 64]

The Rust equivalent, for comparison:

let even_squares: Vec<i32> = (0..10)
    .filter(|x| x % 2 == 0)
    .map(|x| x * x)
    .collect();

Dict Comprehensions

{key_expr: value_expr for item in iterable}:

words   = ["hello", "world", "python"]
lengths = {word: len(word) for word in words}
# {'hello': 5, 'world': 5, 'python': 6}

# invert a dict
original = {"a": 1, "b": 2, "c": 3}
inverted = {v: k for k, v in original.items()}
# {1: 'a', 2: 'b', 3: 'c'}

Set Comprehensions

{expression for item in iterable} — curly braces without a colon:

names        = ["Alice", "Bob", "alice", "ALICE"]
unique_lower = {name.lower() for name in names}
# {'alice', 'bob'}

Generator Expressions

Generator expressions look like list comprehensions but use parentheses. They are lazy: values are computed one at a time as they are consumed, not all upfront:

# list comprehension: entire list built in memory immediately
squares_list = [x ** 2 for x in range(1_000_000)]

# generator expression: one value computed at a time
squares_gen = (x ** 2 for x in range(1_000_000))

Use a generator expression when you only need to iterate once, or when you are passing the result directly to a function that consumes it:

total   = sum(x ** 2 for x in range(1_000_000))     # sum drives the generator
largest = max(len(word) for word in word_list)

When a generator expression is the only argument to a function, the outer parentheses serve double duty as the function call, so you only need one pair.

Nested Comprehensions

A comprehension can contain multiple for clauses, flattening nested structure:

matrix = [[1, 2, 3], [4, 5, 6], [7, 8, 9]]
flat   = [n for row in matrix for n in row]
# [1, 2, 3, 4, 5, 6, 7, 8, 9]

Read left to right: “for each row in matrix, for each n in row, yield n.” The order of the for clauses matches a hand-written nested loop exactly:

flat = []
for row in matrix:
    for n in row:
        flat.append(n)

More than two for clauses becomes hard to read quickly. A regular loop is usually clearer beyond that point.

When to Use Each Form

Form Use when
List comprehension [...] You need the full collection in memory, or will iterate it more than once
Generator expression (...) You iterate once, or pass directly to sum, max, join, etc.
Dict comprehension {k: v ...} Building a mapping from an iterable
Set comprehension {...} Building a deduplicated collection
Regular for loop Multiple statements per iteration, side effects, or early exits

Comprehensions are not just stylistic. Python optimizes them at the bytecode level: a list comprehension is measurably faster than an equivalent for loop with .append() because it avoids looking up the append attribute on every iteration.

Unpacking and Destructuring

Binding multiple names at once

Unpacking assigns multiple names from a sequence in a single statement. It is Python’s counterpart to Rust’s destructuring, though without exhaustive pattern matching. It appears constantly in idiomatic Python code.

Basic Sequence Unpacking

Any iterable on the right-hand side can be unpacked into names on the left:

x, y = (3, 4)
a, b, c = [1, 2, 3]
p, q = "hi"             # strings are iterable too: p = 'h', q = 'i'

The count of names must match the count of values, or you get a ValueError.

Swapping Variables

Unpacking makes the classic swap a one-liner — no temporary variable needed:

a, b = 1, 2
a, b = b, a     # a = 2, b = 1

Python evaluates the entire right-hand side first (producing a tuple (2, 1)), then assigns. There is no aliasing hazard.

Extended Unpacking with *

A starred name captures the remaining elements into a list. It can appear at the start, middle, or end:

first, *rest        = [1, 2, 3, 4, 5]   # first=1,  rest=[2, 3, 4, 5]
*init,  last        = [1, 2, 3, 4, 5]   # init=[1, 2, 3, 4],  last=5
a,     *middle, b   = [1, 2, 3, 4, 5]   # a=1,  middle=[2, 3, 4],  b=5

The starred name always receives a list, even when zero elements match:

first, *rest = [42]   # first=42, rest=[]

Unpacking in for Loops

Any iterable of sequences can be unpacked directly in the loop variable:

pairs = [(1, "one"), (2, "two"), (3, "three")]

for number, word in pairs:
    print(number, word)

Dict .items() is the most common real-world case:

data = {"a": 1, "b": 2}

for key, value in data.items():
    print(key, value)

This is also why enumerate and zip feel so natural — their output is a sequence of tuples, designed to be unpacked:

for i, item in enumerate(["a", "b", "c"]):
    print(i, item)                          # 0 a, 1 b, 2 c

for left, right in zip([1, 2, 3], ["x", "y", "z"]):
    print(left, right)                      # 1 x, 2 y, 3 z

Nested Unpacking

Unpacking mirrors the shape of the data, however deeply nested:

(x1, y1), (x2, y2) = (0, 0), (3, 4)

segments = [((0, 0), (1, 1)), ((2, 2), (3, 3))]

for (x1, y1), (x2, y2) in segments:
    print(x1, y1, "->", x2, y2)

Unpacking into Function Calls

* and ** unpack sequences and dicts into positional and keyword arguments at the call site. This is the mirror image of *args / **kwargs in a function definition:

def add(a, b, c):
    return a + b + c

args   = [1, 2, 3]
kwargs = {"a": 1, "b": 2, "c": 3}

add(*args)     # same as add(1, 2, 3)
add(**kwargs)  # same as add(a=1, b=2, c=3)

Both can be mixed with literal arguments:

def f(a, b, c, d):
    ...

f(1, *[2, 3], d=4)   # a=1, b=2, c=3, d=4

This is useful when arguments are built programmatically — for example, when forwarding arguments from one function to another, or constructing a call from a config dict.

The Walrus Operator :=

The walrus operator assigns a value and simultaneously returns it as an expression. It lets you assign inside a condition or comprehension without a separate statement:

# without walrus: the readline call must be duplicated
line = file.readline()
while line:
    process(line)
    line = file.readline()

# with walrus: assign and test in one expression
while line := file.readline():
    process(line)

In comprehensions it avoids computing an expensive value twice:

# expensive(x) is called twice for every element that passes the filter
results = [expensive(x) for x in data if expensive(x) > 10]

# walrus computes it once and reuses the result
results = [y for x in data if (y := expensive(x)) > 10]
NoteWhere := is allowed

The walrus operator is intentionally restricted to expression contexts: while conditions, if conditions, and comprehension filters. It cannot be used as a plain statement — that is what = is for. The restriction prevents it from replacing regular assignment, keeping the two visually distinct.

Reference Semantics and Memory

This section is the Python equivalent of the Memory Management section in the Rust chapter. Python hides the mechanics, but understanding them prevents a whole class of bugs that are otherwise very hard to track down.

Everything Is a Heap Object

In C, int x = 5 allocates four bytes on the stack. In Python, x = 5 creates an int object on the heap and binds the name x to it. Every Python value — integers, strings, functions, classes — lives on the heap. The interpreter manages all of it.

The variable x is not a box containing a value. It is a name in a namespace dictionary that points to an object. This distinction matters.

Assignment Copies the Reference, Not the Object

a = [1, 2, 3]
b = a           # b points to the SAME list object
b.append(4)

print(a)        # [1, 2, 3, 4] — a is affected

This is not a copy. Both a and b are names bound to the same list. The list has one identity and two names. Modifying the list through either name modifies the one shared object.

For comparison, the Rust equivalent would be a compile error: the borrow checker would reject two mutable references to the same data. Python has no such check; the aliasing is silent.

You can inspect identity with id() and the is operator:

a = [1, 2, 3]
b = a
print(id(a) == id(b))   # True: same object in memory
print(a is b)           # True: `is` checks identity, not equality
print(a == b)           # True: `==` checks value equality

Mutable vs Immutable Types

The behavior of assignment depends on whether the type is mutable or immutable.

Immutable types: int, float, bool, str, tuple, frozenset, bytes. You cannot change the object after it is created. Any operation that looks like modification creates a new object instead:

x = "hello"
y = x
x += " world"   # x is now bound to a NEW str object

print(y)        # "hello": y still points to the original

Mutable types: list, dict, set, bytearray, and most user-defined objects. The object can be modified in place, and all references see the change:

a = {"key": 1}
b = a
a["key"] = 99

print(b)        # {"key": 99}: same dict, one change visible through both names

This asymmetry is one of the most common sources of bugs for programmers new to Python. The rule to remember: assignment always copies the reference. For immutable types, this is safe because the object cannot change. For mutable types, it creates an alias.

Copying Mutable Objects

Three levels of copy:

import copy

original = [[1, 2], [3, 4]]

# 1. Alias: no copy at all
alias = original

# 2. Shallow copy: new container, same inner objects
shallow = original.copy()         # or list(original) or original[:]
shallow.append([5, 6])            # original unaffected: new container
shallow[0].append(99)             # original[0] IS affected: inner list is shared

# 3. Deep copy: new everything, recursively
deep = copy.deepcopy(original)
deep[0].append(99)                # original unaffected at every level

A shallow copy creates a new container but does not copy the objects inside it. If those objects are themselves mutable, the shallow copy shares them with the original. Deep copy recurses through the entire structure. For nested mutable data, the difference matters.

Identity, Equality, and Interning

is tests whether two names point to the same object. == tests whether two objects have equal values. They usually agree, but not always:

a = [1, 2, 3]
b = [1, 2, 3]

print(a == b)   # True: equal values
print(a is b)   # False: two distinct list objects

One exception worth knowing: CPython caches small integers (typically -5 through 256) and short strings that look like identifiers. These objects are reused rather than re-created, so is may return True even for separately created values in that range:

x = 42
y = 42
print(x is y)   # True in CPython: the integer 42 is cached

x = 1000
y = 1000
print(x is y)   # False: large integers are not cached

Never rely on this caching behavior. Use == to compare values and is only when you specifically mean to test object identity (most commonly: if x is None).

How Lists Allocate Memory

Python lists are dynamic arrays, similar to Vec<T> in Rust or std::vector in C++. They maintain an internal buffer that is larger than the current length, so that append is fast most of the time.

When the buffer is full and you append another element, Python reallocates: it allocates a new, larger buffer, copies the elements over, and frees the old one. The new capacity follows a growth formula that adds roughly one-eighth of the current size (plus a small constant for small lists). This gives a growth factor approaching 1.125 for large lists, smaller than Rust’s factor of 2 but still amortized O(1) per append.

You can observe the reallocation steps directly:

import sys

a = []
prev_size = 0
for i in range(20):
    a.append(i)
    size = sys.getsizeof(a)
    if size != prev_size:
        print(f"len={len(a):2d}  buffer size: {size} bytes")
        prev_size = size

The buffer grows in steps, not on every append. You pay for reallocation infrequently. If you know the final size in advance, a list comprehension avoids all intermediate reallocations by building the list in one pass:

# repeated append: may reallocate several times
result = []
for i in range(10000):
    result.append(compute(i))

# comprehension: single allocation, then fill
result = [compute(i) for i in range(10000)]

Dict and Set Internals

Dicts and sets are hash tables. Lookup, insertion, and deletion are O(1) average case. The hash table resizes when the load factor exceeds roughly two-thirds capacity, at which point it doubles in size. As with lists, this reallocation is amortized over many insertions.

Python dicts preserve insertion order as of Python 3.7. This is guaranteed by the language specification.

Reference Counting and Garbage Collection

CPython tracks a reference count on every object: the number of names and data structures that point to it. When the count drops to zero, the object is freed immediately and deterministically. This is similar in spirit to RAII in C++ or Rust’s ownership system, but managed automatically by the runtime.

import sys

a = [1, 2, 3]
b = a                       # reference count is now 2
print(sys.getrefcount(a))   # 3 (getrefcount itself adds one temporarily)

del a                       # count drops to 1: not freed yet
del b                       # count drops to 0: freed immediately

Reference counting has one weakness: cycles. If object A holds a reference to object B, and B holds a reference back to A, neither count ever reaches zero, even if nothing else refers to them. CPython includes a separate cycle garbage collector that periodically detects and clears these cycles. It runs automatically; you rarely need to think about it.

The practical implication of reference counting: CPython’s memory release is more predictable than a generational GC. Objects are usually freed as soon as they go out of scope, not at some future collection point. This matters for resources like file handles and network connections: the with statement exploits this predictability.

with open("data.txt") as f:
    content = f.read()
# file is closed here, immediately, when the block exits

The Global Interpreter Lock

CPython has a Global Interpreter Lock (GIL): only one thread executes Python bytecode at a time. This makes CPython’s internals thread-safe without fine-grained locking, but it prevents true CPU parallelism across threads.

The practical implications:

  • I/O-bound work: threads work well. The GIL is released during I/O operations, so multiple threads can overlap network calls, file reads, and so on without blocking each other.
  • CPU-bound work: use multiprocessing (separate processes, each with their own GIL) or offload to C extensions like NumPy, which release the GIL for their inner loops.

Python 3.13 introduced an experimental free-threaded build (no GIL) as an opt-in option. It is not yet the default.

Memory-Efficient Patterns

Generators

A list comprehension builds the entire result in memory at once. For large or infinite sequences, a generator computes values lazily, one at a time:

# list comprehension: allocates all 10 million integers immediately
big_list = [x * 2 for x in range(10_000_000)]

# generator expression: allocates one integer at a time
big_gen = (x * 2 for x in range(10_000_000))

# generator function: same behavior, more control
def doubled(n):
    for i in range(n):
        yield i * 2         # suspends here, resumes on next iteration

for value in doubled(10_000_000):
    process(value)          # only one integer in memory at a time

The yield keyword turns a function into a generator. Its local state is suspended between calls and resumed on the next iteration. Generators are the idiomatic way to handle large data streams, infinite sequences, and processing pipelines without holding everything in memory.

Functions and Closures

First-Class Functions

Functions are objects. They can be passed, returned, and stored:

    return func(value)

    return x * 2

Closures

A nested function can capture variables from its enclosing scope:

def make_multiplier(n):
    def multiply(x):
        return x * n   # captures n from the enclosing scope
    return multiply

triple = make_multiplier(3)
triple(7)   # 21

This is the same concept as Rust closures or C++ lambdas with captures. Unlike Rust, Python does not distinguish capture by reference from capture by value: closures always capture a reference to the enclosing scope’s variable binding.

Decorators

A decorator is a function that wraps another function. The @ syntax is shorthand for applying the wrapper:

def log_calls(func):
    def wrapper(*args, **kwargs):
        print(f"calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

@log_calls
def add(a, b):
    return a + b

# equivalent to: add = log_calls(add)
add(2, 3)   # prints "calling add", returns 5

Decorators are used throughout web frameworks, testing libraries, and the standard library (@staticmethod, @classmethod, @property).

Classes and Objects

class Point:
    def __init__(self, x, y):           # constructor
        self.x = x                      # instance attributes
        self.y = y

    def distance_from_origin(self):
        return (self.x**2 + self.y**2) ** 0.5

    def __repr__(self):                 # string representation for debugging
        return f"Point({self.x}, {self.y})"

    def __eq__(self, other):            # equality operator
        return self.x == other.x and self.y == other.y

p = Point(3, 4)

print(p)                          # Point(3, 4)

self is explicit: Python does not inject it invisibly. Every instance method takes self as its first parameter, giving you a clear signal that the method operates on the object.

Dunder Methods

The object protocol

Dunder methods (double underscore on both sides: __init__, __len__, __iter__, etc.) are how Python objects hook into the interpreter’s built-in operations. When you write len(obj), Python calls obj.__len__(). When you write for x in obj, Python calls obj.__iter__(). When you write with obj, Python calls obj.__enter__(). The built-in syntax is a thin layer over these methods.

This is what duck typing means at the implementation level: an object is a sequence if it has __len__ and __getitem__. An object is iterable if it has __iter__. The type hierarchy matters far less than the protocol an object implements.

Operator Overloading

Arithmetic and comparison operators map directly to dunders:

class Vector:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __add__(self, other):
        return Vector(self.x + other.x, self.y + other.y)

    def __mul__(self, scalar):
        return Vector(self.x * scalar, self.y * scalar)

    def __rmul__(self, scalar):
        return self.__mul__(scalar)   # makes 3 * v work, not just v * 3

    def __eq__(self, other):
        return self.x == other.x and self.y == other.y

    def __repr__(self):
        return f"Vector({self.x}, {self.y})"

print(v + Vector(3, 4))   # Vector(4, 6)
print(v * 3)              # Vector(3, 6)
print(3 * v)              # Vector(3, 6) — via __rmul__

The Container Protocol

Implement these to make an object behave like a sequence or mapping:

class Countdown:
    def __init__(self, start):
        self.data = list(range(start, -1, -1))

    def __len__(self):
        return len(self.data)       # len(obj)

    def __getitem__(self, index):
        return self.data[index]     # obj[i] and slicing

    def __contains__(self, value):
        return value in self.data   # `in` operator

c = Countdown(5)
print(len(c))     # 6
print(c[0])       # 5
print(3 in c)     # True

If you implement __getitem__ with integer indices starting at 0, Python will allow for x in obj to work automatically by repeatedly calling __getitem__ with increasing indices. It is cleaner to implement __iter__ explicitly.

The Iteration Protocol

for x in obj calls iter(obj) to get an iterator, then calls next() on it repeatedly until StopIteration is raised. You can implement this in two ways.

Option 1: The object is its own iterator. Implement both __iter__ (returns self) and __next__ (returns the next value or raises StopIteration):

class CountUp:
    def __init__(self, limit):
        self.limit = limit
        self.current = 0

    def __iter__(self):
        return self

    def __next__(self):
        if self.current >= self.limit:
            raise StopIteration
        value = self.current
        self.current += 1
        return value

list(CountUp(4))   # [0, 1, 2, 3]

The downside: iterating the same object twice picks up where the first left off. The state is shared.

Option 2: Return a fresh iterator from __iter__. The collection and the iterator are separate objects, so multiple independent loops work correctly:

class NumberRange:
    def __init__(self, start, stop):
        self.start = start
        self.stop = stop

    def __iter__(self):
        return iter(range(self.start, self.stop))   # fresh iterator each time

For complex iteration logic, a generator inside __iter__ is the most readable approach:

class EvenNumbers:
    def __init__(self, limit):
        self.limit = limit

    def __iter__(self):
        for n in range(0, self.limit, 2):
            yield n   # __iter__ becomes a generator; no __next__ needed

Context Managers from the Inside

with obj as x: calls obj.__enter__() on entry (binding the return value to x) and obj.__exit__() on exit, whether or not an exception was raised:

import time

class Timer:
    def __enter__(self):
        self.start = time.perf_counter()
        return self                         # this is what `as t` receives

    def __exit__(self, exc_type, exc_val, exc_tb):
        elapsed = time.perf_counter() - self.start
        print(f"elapsed: {elapsed:.4f}s")
        return False                        # False: let exceptions propagate

with Timer():
    sum(range(1_000_000))
# elapsed: 0.0187s

__exit__ receives the exception type, value, and traceback if an exception occurred, or None for all three if the block exited cleanly. Returning True suppresses the exception; returning False or None lets it propagate.

    sum(range(1_000_000))

@property, @classmethod, and @staticmethod

These three decorators control how a method relates to its class and instances.

@property

@property turns a method into a computed attribute. From the outside it looks like a plain attribute access; under the hood a method runs:

import math

class Circle:
    def __init__(self, radius):
        self.radius = radius

    @property
    def area(self):
        return math.pi * self.radius ** 2

c = Circle(5)
print(c.area)   # 78.54... — no parentheses; looks like an attribute, runs a method

No parentheses on the call side. This is more than cosmetic: you can start with a plain attribute and later replace it with a computed property without changing any code that uses it. Callers never know the difference.

To allow setting and deleting, add a setter with the same name:

class Temperature:
    def __init__(self, celsius):
        self._celsius = celsius         # _celsius is the private backing attribute

    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("temperature below absolute zero")
        self._celsius = value

    @property
    def fahrenheit(self):               # read-only: no setter defined
        return self._celsius * 9 / 5 + 32

t = Temperature(25)
print(t.fahrenheit)   # 77.0
t.celsius = 100       # calls the setter
t.celsius = -300      # ValueError
t.fahrenheit = 50     # AttributeError: no setter defined

A read-only property (getter but no setter) raises AttributeError on assignment. This is a clean way to expose a computed value that callers should never set directly.

@classmethod

A class method receives the class as its first argument (cls) instead of the instance (self). It is called on the class, not on an instance. The most common use is alternative constructors:

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    @classmethod
    def from_tuple(cls, t):
        return cls(t[0], t[1])          # cls(...) instead of Point(...) — important for subclasses

    @classmethod
    def origin(cls):
        return cls(0, 0)

    def __repr__(self):
        return f"Point({self.x}, {self.y})"

p1 = Point.from_tuple((3, 4))   # Point(3, 4)
p2 = Point.origin()             # Point(0, 0)

Using cls(...) rather than Point(...) matters for inheritance: if ColorPoint subclasses Point, then ColorPoint.from_tuple(...) correctly returns a ColorPoint, not a Point.

@staticmethod

A static method has no implicit first argument — no self, no cls. It is an ordinary function that lives inside the class for organizational reasons:

class MathUtils:
    @staticmethod
    def clamp(value, lo, hi):
        return max(lo, min(hi, value))

MathUtils.clamp(15, 0, 10)   # 10

A static method could equally be a module-level function. You put it in the class when it is logically part of the class’s responsibility but does not need to read or modify any instance or class state.

super() and MRO

Calling the Parent with super()

super() returns a proxy that delegates method calls to the next class in the inheritance chain. The most common use is calling the parent’s __init__ to ensure it runs before you add subclass-specific state:

class Animal:
    def __init__(self, name):
        self.name = name

    def describe(self):
        return f"I am {self.name}"

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)      # run Animal.__init__ first
        self.breed = breed          # then add Dog-specific state

    def describe(self):
        return f"{super().describe()}, a {self.breed}"

d = Dog("Rex", "Labrador")
print(d.describe())   # I am Rex, a Labrador

You could call Animal.__init__(self, name) directly, but super() is better for two reasons: it automatically follows the correct chain if the hierarchy changes, and it is required for correct behavior in multiple inheritance.

Method Resolution Order

MRO

When Python looks up a method, it searches the class and its bases in a specific order called the MRO (Method Resolution Order). For single inheritance this is simply the chain from the class up to object. For multiple inheritance it is computed by the C3 linearization algorithm.

You can inspect the MRO of any class:

print(Dog.__mro__)
# (<class '__main__.Dog'>, <class '__main__.Animal'>, <class 'object'>)

Every class in Python ultimately inherits from object, which is why __repr__, __str__, and others always exist as a fallback.

Multiple Inheritance and the Diamond Problem

Multiple inheritance means a class can list more than one parent:

class A:
    def greet(self):
        return "A"

class B(A):
    def greet(self):
        return f"B, {super().greet()}"

class C(A):
    def greet(self):
        return f"C, {super().greet()}"

class D(B, C):
    def greet(self):
        return f"D, {super().greet()}"

This is the classic diamond problem: D inherits from B and C, which both inherit from A. Without careful ordering, A.greet() could be called twice.

Python’s MRO ensures each class in the hierarchy appears exactly once in the lookup order:

print(D.__mro__)
# (<class 'D'>, <class 'B'>, <class 'C'>, <class 'A'>, <class 'object'>)

print(D().greet())
# D, B, C, A

A.greet() is called exactly once. The C3 algorithm guarantees three properties:

  1. A class always appears before its parents.
  2. Parents appear in the left-to-right order given in the class definition.
  3. Each class appears only once.

Cooperative super()

For super() to work correctly across the full MRO, every class in the chain must call super() — including the ones in the middle. This is called cooperative multiple inheritance:

class LoggingMixin:
    def __init__(self, *args, **kwargs):
        print(f"creating {self.__class__.__name__}")
        super().__init__(*args, **kwargs)   # pass everything up the chain

class TrackedDog(LoggingMixin, Dog):
    pass

d = TrackedDog("Spot", "Beagle")
# creating TrackedDog
# d.name == "Spot", d.breed == "Beagle"

If any class in the chain calls its parent directly (Animal.__init__(self, ...)) instead of using super(), it short-circuits the MRO and classes further along are skipped.

NoteThe rule of thumb

If you are writing a class that might participate in multiple inheritance, always use super() rather than naming the parent class directly. For classes you are certain will never be mixed in, either style works, but super() is the safer habit.

No Access Control

Python has no private or protected keywords. Convention uses a leading underscore to signal “internal: do not use this from outside”:

Error Handling

Python uses exceptions. Errors are raised with raise and caught with try / except:

def divide(a, b):
    if b == 0:
        raise ValueError("cannot divide by zero")
    return a / b

try:
    result = divide(10, 0)
except ValueError as e:
    print(f"error: {e}")
except ZeroDivisionError:
    print("division by zero")
else:
    print(f"result: {result}")  # runs only if no exception was raised
finally:
    print("always runs")        # cleanup: always executes, exception or not

Exceptions are classes. The standard library defines a hierarchy; catching a base class catches all its subclasses:

try:
    f = open("data.txt")
except OSError as e:            # catches FileNotFoundError, PermissionError, etc.
    print(e)

Context Managers

The with statement ensures cleanup happens even if an exception is raised. It is Python’s answer to RAII:

with open("data.txt") as f:
    content = f.read()
# file is closed here, regardless of whether an exception occurred

# multiple resources in one statement
with open("input.txt") as src, open("output.txt", "w") as dst:
    dst.write(src.read())

Any object implementing __enter__ and __exit__ can be used as a context manager. The standard library provides them for files, locks, database connections, and more.

Concurrency

Doing more than one thing at a time

Python offers three concurrency models, each suited to a different class of problem:

Model Module Best for
Async / await asyncio Many simultaneous I/O operations: network calls, database queries
Threads threading I/O-bound work using blocking libraries that have no async support
Processes multiprocessing CPU-bound computation: number crunching, image processing

The GIL means threads cannot run Python bytecode in parallel. Both threads and async are I/O concurrency tools. For true CPU parallelism, you need separate processes or native extensions like NumPy that release the GIL internally.

async / await

Cooperative concurrency on a single thread

async/await gives you concurrency without threads. The mechanism is a single-threaded event loop that manages a collection of coroutines. A coroutine is a function that can pause itself at an await expression, hand control back to the event loop, and be resumed when the awaited operation is ready.

The key insight: no thread switching happens. There is one thread. Coroutines voluntarily yield control at every await. This eliminates entire classes of threading bugs (race conditions, deadlocks) at the cost of requiring you to explicitly mark every suspension point.

If you have written async Rust, the mental model is identical. The practical difference is that Python’s event loop (asyncio) is in the standard library and is always single-threaded by default.

The Basic Pattern

import asyncio

async def fetch_data(name: str) -> str:
    print(f"starting: {name}")
    await asyncio.sleep(1)          # simulates a network call; suspends here
    print(f"finished: {name}")
    return f"data from {name}"

async def main():
    result = await fetch_data("example")
    print(result)

asyncio.run(main())                 # creates the event loop, runs main, closes loop

async def defines a coroutine function. Calling it does not execute it — it returns a coroutine object. await runs it and suspends the caller until it completes. asyncio.run() is the program’s entry point into async: it creates the event loop, runs the top-level coroutine to completion, then tears down the loop.

Concurrency with gather

Awaiting coroutines one at a time is sequential, not concurrent. To run multiple coroutines simultaneously, use asyncio.gather:

import asyncio

async def fetch(name: str, delay: float) -> str:
    await asyncio.sleep(delay)
    return f"{name} done"

async def main():
    # all three run concurrently: total time ~2s, not ~4s
    results = await asyncio.gather(
        fetch("a", 2.0),
        fetch("b", 1.5),
        fetch("c", 0.5),
    )
    print(results)   # ['a done', 'b done', 'c done']

asyncio.run(main())

gather schedules all coroutines immediately, then waits for every one to finish. While one is suspended at await asyncio.sleep, the others can run. Results are returned in argument order regardless of which finished first.

Tasks

asyncio.create_task schedules a coroutine to start running immediately without waiting for it. This is useful when you want to fire off background work and continue doing other things:

async def main():
    task = asyncio.create_task(fetch("background", 2.0))

    # event loop runs the task in the background while we do other work
    await asyncio.sleep(0.5)
    print("did other work")

    result = await task     # now wait for the background task to finish
    print(result)

The distinction from gather: create_task gives you a handle to one coroutine so you can await it on your own schedule. gather groups coroutines and waits for all of them together.

async with and async for

Context managers and iterators can also be async, allowing I/O to happen inside them without blocking the event loop:

# async context manager: resources that require async setup or teardown
async with aiohttp.ClientSession() as session:
    async with session.get("http://example.com") as response:
        text = await response.text()

# async for: consuming an async source line by line
async for line in async_stream:
    process(line)

Any class implementing __aenter__ and __aexit__ is an async context manager. Any class implementing __aiter__ and __anext__ is an async iterable. These mirror the synchronous dunder methods exactly.

The Propagation Rule

You cannot await inside a regular (non-async) function. Async calls can only come from other async functions or from asyncio.run at the top level:

def regular():
    await something()   # SyntaxError

async def async_fn():
    await something()   # fine

This creates a hard boundary: once you use async in one part of a program, it tends to propagate upward through every caller. Plan for this before adopting async in a large codebase.

NoteAsync requires async-native libraries

asyncio provides the runtime, but performing I/O requires async-native libraries. Using a synchronous library (like requests) inside an async function blocks the entire event loop for the duration of the call, defeating the purpose. The async equivalents: httpx or aiohttp for HTTP, aiosqlite for SQLite, asyncpg for PostgreSQL.

Threads

For I/O-bound code that uses libraries without async support, threads are the practical choice. The GIL is released during blocking I/O, so threads genuinely overlap on network and file work:

import threading
import requests

def fetch(url):
    response = requests.get(url)    # blocking — fine in a thread
    print(response.status_code)

urls = ["http://example.com", "http://example.org", "http://example.net"]
threads = [threading.Thread(target=fetch, args=(url,)) for url in urls]
for t in threads: t.start()
for t in threads: t.join()          # wait for all to complete

Threads share memory, so shared mutable state requires locks (threading.Lock). This is the classic hazard: threads are easy to start and easy to get wrong.

Processes

For CPU-bound work, separate processes bypass the GIL entirely. Each process has its own Python interpreter and its own memory space:

from multiprocessing import Pool

def compute(x):
    return x * x

with Pool(processes=4) as pool:
    results = pool.map(compute, range(100))   # distributed across 4 processes

Pool.map distributes work across processes and collects results. The overhead of spawning processes and serializing data between them is real. This pays off only when the per-item computation is substantial.

Choosing a Model

Situation Use
Many simultaneous network/database calls async / await
Blocking library with no async support, I/O-bound threading
CPU-bound computation multiprocessing
Wrapping a C/Rust extension that releases the GIL Either threads or processes

Performance

See Python Performance Analysis for a deeper dive.

Python is often criticized for being “slow.” For a systems programmer used to C or Rust, this is true: Python’s abstraction layer is thick. Every operation involves dynamic dispatch and object management overhead. However, Python is rarely the bottleneck for I/O-bound tasks, and for CPU-bound tasks, it provides excellent facilities for offloading work to compiled code.

The Cost of Dynamic Dispatch

In C, a + b might be a single assembly instruction. In Python, it is a method lookup:

  • Check if a has an __add__ method.
  • If not, check if b has an __radd__ method.
  • Execute the method, which involves type checking and creating a new heap-allocated object for the result.

This overhead exists for every variable access, function call, and attribute lookup. This is why Python is 10x to 100x slower than C for raw numerical loops.

What Improves Performance

  • Use built-in functions. Functions like sum(), min(), max(), any(), and all() are implemented in C. They iterate over data at C speed.
  • List comprehensions. These are optimized at the bytecode level. They are faster than a for loop that calls .append() because they avoid the repeated lookup of the append method on the list object.
  • Local variables. Accessing a local variable is faster than a global variable. In a tight loop, binding a global function or method to a local name can yield measurable gains.
  • Generators. While not necessarily faster in CPU time, they are significantly more memory-efficient for large datasets, preventing the performance degradation caused by memory pressure.

What Hurts Performance

  • The GIL. The Global Interpreter Lock (GIL) prevents multiple threads from executing Python bytecode simultaneously. For CPU-bound work, threads will actually slow down your program due to context switching overhead without any parallelism gain.
  • String concatenation. Since strings are immutable, s += "part" creates a new string object and copies the old content. In a loop, this is \(O(n^2)\). Always use "".join(list_of_parts).
  • Manual loops. Writing while i < len(lst): x = lst[i] is significantly slower than for x in lst:. Always prefer for x in lst:.

Interfacing with C and Rust

The “Pythonic” way to handle performance-critical code is to write it in C, C++, or Rust and call it from Python. This is how libraries like NumPy, TensorFlow, and Pydantic achieve near-native speed.

Extension Modules

You can write modules in C using the Python C API. This gives you full control but is verbose and error-prone. Cython is a popular alternative: it allows you to write Python-like code with optional type declarations, which it then compiles to C.

FFI (Foreign Function Interface)

Libraries like ctypes (standard library) and cffi allow you to load shared libraries (.so, .dll, .dylib) and call functions within them directly from Python. This is ideal for wrapping existing C libraries.

The Rust Bridge

PyO3 is the modern standard for writing Python extensions in Rust. It provides a safe, idiomatic interface that handles the conversion between Python objects and Rust types, allowing you to bypass the GIL safely and leverage Rust’s performance.

Efficient Iteration and Data Structures

Choosing the right structure is the most impactful performance decision.

Task Data Structure Efficiency
Membership testing set or dict \(O(1)\)
Membership testing list \(O(n)\)
Frequent insertions at start collections.deque \(O(1)\)
Frequent insertions at start list \(O(n)\)

The Power of Vectorization

If you are doing heavy math, do not iterate at all. Use NumPy.

import numpy as np

def double_elements(my_list: list[float]) -> np.ndarray:
  """Vectorized doubling of list elements."""
  arr: np.ndarray = np.array(my_list)
  return arr * 2

Vectorized operations apply a single instruction to multiple data points (SIMD) at the C level, bypassing the Python interpreter entirely for the bulk of the work.

Standard Library Highlights

The batteries that are already included

Python’s standard library is enormous. These are the modules you will reach for most often in day-to-day work, beyond the built-in types covered earlier.

pathlib

Object-oriented file paths

pathlib is the modern standard for working with file system paths. Rather than manipulating path strings with os.path.join(...), you work with Path objects that understand the structure of a path and expose every common operation as a method.

Rust programmers will recognize the conceptual split between Path (a borrowed view of a path) and PathBuf (an owned, growable path). In Python, pathlib.Path always owns its data — the equivalent of PathBuf.

Creating Paths

from pathlib import Path

p = Path("/usr/local/bin")      # absolute path
p = Path("data/output.csv")    # relative path
p = Path.cwd()                  # current working directory
p = Path.home()                 # home directory: /Users/alice or similar

The / operator joins path components. It is clean, readable, and cross-platform:

base   = Path("/home/alice")
config = base / ".config" / "app" / "settings.toml"
# /home/alice/.config/app/settings.toml

Inspecting a Path

p = Path("/home/alice/data/report.final.csv")

p.name      # "report.final.csv"    — filename with all extensions
p.stem      # "report.final"        — filename without last extension
p.suffix    # ".csv"                — last extension only
p.suffixes  # [".final", ".csv"]    — all extensions
p.parent    # /home/alice/data      — containing directory
p.parts     # ('/', 'home', 'alice', 'data', 'report.final.csv')

Checking What Exists

p = Path("data/report.csv")

p.exists()    # True if the path exists as anything
p.is_file()   # True if it exists and is a regular file
p.is_dir()    # True if it exists and is a directory
p.stat()      # os.stat_result: size, mtime, permissions, etc.

Reading and Writing

For files small enough to fit in memory, Path provides one-liners:

content = Path("notes.txt").read_text(encoding="utf-8")
Path("output.txt").write_text("hello\n", encoding="utf-8")

data = Path("image.png").read_bytes()
Path("copy.png").write_bytes(data)

For large files or line-by-line work, use p.open():

with p.open(encoding="utf-8") as f:
    for line in f:
        process(line)

p.open(...) is equivalent to open(str(p), ...) but keeps the path as a Path object throughout rather than converting to a string.

Creating Directories and Files

Path("output/results").mkdir(parents=True, exist_ok=True)
# parents=True:  creates any missing parent directories (like mkdir -p)
# exist_ok=True: does not raise an error if the directory already exists

Path("output/results/data.csv").touch()   # create empty file or update timestamp

Listing and Globbing

src = Path("src")

for item in src.iterdir():       # immediate contents
    print(item)

for f in src.glob("*.py"):       # pattern match in this directory
    print(f)

for f in src.rglob("*.py"):     # recursive: all .py files in the entire tree
    print(f)

Modifying and Resolving Paths

# resolve: canonicalize — expand .., resolve symlinks, make absolute
absolute = Path("../data/report.csv").resolve()

# relative_to: express one path relative to another
full = Path("/home/alice/project/src/main.py")
rel  = full.relative_to(Path("/home/alice/project"))   # src/main.py

# with_suffix / with_name: derive a modified path without mutating
p       = Path("archive.tar.gz")
p.with_suffix(".bz2")             # archive.tar.bz2
p.with_name("backup.tar.gz")      # backup.tar.gz
Noteos.path vs pathlib

The older os.path module manipulates paths as plain strings: os.path.join("/home", "alice", "file.txt"). You will see it in older codebases. For new code, prefer pathlib — the / operator, the method-based API, and the clean separation of name parts make it substantially less error-prone.

collections

Specialized container types

The collections module provides container types that extend the built-in dict, list, and tuple for patterns that would otherwise require custom boilerplate.

Counter

Counter counts occurrences of elements. It is a dict subclass mapping each element to its count:

from collections import Counter

words = ["apple", "banana", "apple", "cherry", "banana", "apple"]
c = Counter(words)
# Counter({'apple': 3, 'banana': 2, 'cherry': 1})

c["apple"]        # 3
c["missing"]      # 0 — no KeyError for absent keys

c.most_common(2)  # [('apple', 3), ('banana', 2)] — top n by count

Counters support arithmetic, which makes them useful for comparing or combining frequency distributions:

a = Counter(["x", "x", "y"])
b = Counter(["x", "z"])

a + b   # Counter({'x': 3, 'y': 1, 'z': 1})  — add counts
a - b   # Counter({'x': 1, 'y': 1})           — subtract, drop zeros and negatives
a & b   # Counter({'x': 1})                    — minimum of each count
a | b   # Counter({'x': 2, 'y': 1, 'z': 1})  — maximum of each count

Rust equivalent: a HashMap<T, usize> built manually with .entry().and_modify().or_insert(0).

defaultdict

defaultdict is a dict subclass that calls a factory function to supply a default value for any missing key, instead of raising KeyError:

from collections import defaultdict

groups = defaultdict(list)   # factory is any zero-argument callable

for word in ["apple", "banana", "avocado", "blueberry", "apricot"]:
    groups[word[0]].append(word)   # no KeyError on first access for a new key

# {'a': ['apple', 'avocado', 'apricot'], 'b': ['banana', 'blueberry']}

Without defaultdict, grouping requires an explicit guard: if key not in d: d[key] = []. The factory is called only when a missing key is accessed, so existing keys are unaffected.

Factory Default value
list []
set set()
int 0
dict {}
lambda: "N/A" "N/A"

Rust equivalent: map.entry(key).or_insert_with(Vec::new).

namedtuple

namedtuple creates a lightweight immutable record type with named fields. It is a tuple subclass, so it uses the same memory as a plain tuple while also supporting attribute access:

from collections import namedtuple

Point = namedtuple("Point", ["x", "y"])

p = Point(3.0, 4.0)
print(p.x, p.y)   # 3.0 4.0  — attribute access
print(p[0])       # 3.0       — index access still works
x, y = p          # unpacking works too

p._replace(x=10)  # Point(10.0, 4.0) — returns a new instance; original unchanged
p._asdict()       # {'x': 3.0, 'y': 4.0}

For new code, @dataclass(frozen=True) is often more flexible. namedtuple remains common in older codebases and in performance-sensitive code where its tuple memory layout matters.

deque

deque (double-ended queue) supports O(1) append and pop from both ends. A regular list’s insert(0, x) and pop(0) are O(n) because they shift all elements; deque avoids that:

from collections import deque

d = deque([1, 2, 3])
d.append(4)       # right end: [1, 2, 3, 4]
d.appendleft(0)   # left end:  [0, 1, 2, 3, 4]
d.pop()           # remove from right: returns 4
d.popleft()       # remove from left:  returns 0

d.rotate(1)       # shift right by 1: [3, 1, 2]
d.rotate(-1)      # shift left by 1:  [1, 2, 3]

deque accepts a maxlen argument. When the deque is full, appending to one end automatically drops from the other — a fixed-size sliding window:

recent = deque(maxlen=5)
for event in stream:
    recent.append(event)   # always holds the last 5 events

Rust equivalent: std::collections::VecDeque<T>.

functools

Higher-order function utilities

functools provides tools for working with functions as values: caching results, fixing arguments, reducing sequences, and writing well-behaved decorators.

@lru_cache and @cache

@lru_cache memoizes a function: the first time it is called with a given set of arguments the result is computed and stored. Subsequent calls with the same arguments return the cached value immediately:

from functools import lru_cache

@lru_cache(maxsize=128)
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

fibonacci(35)              # fast: each subproblem is computed only once
fibonacci.cache_info()     # CacheInfo(hits=33, misses=36, maxsize=128, currsize=36)
fibonacci.cache_clear()    # evict all cached entries

@cache (Python 3.9+) is the same with no size limit — equivalent to @lru_cache(maxsize=None):

from functools import cache

@cache
def factorial(n):
    return 1 if n == 0 else n * factorial(n - 1)

Both require that all arguments be hashable. Lists and dicts cannot be passed to a cached function because they cannot be used as dict keys.

partial

partial creates a new callable with some arguments pre-filled, without writing a wrapper function manually:

from functools import partial

def power(base, exponent):
    return base ** exponent

square = partial(power, exponent=2)
cube   = partial(power, exponent=3)

square(5)   # 25
cube(3)     # 27

This is useful when an API expects a specific signature but you have a more general function you want to specialize:

from functools import partial
from pathlib import Path

# pre-fill the base directory so callers only supply the relative part
data_path = partial(Path("/data").joinpath)
data_path("reports", "q4.csv")   # /data/reports/q4.csv

Rust equivalent: a closure that captures some arguments — let square = |x| power(x, 2);.

reduce

reduce applies a two-argument function cumulatively to a sequence, collapsing it to a single accumulated value. This is the same concept as .fold() in Rust iterators:

from functools import reduce

numbers = [1, 2, 3, 4, 5]

product = reduce(lambda acc, x: acc * x, numbers)       # 120
product = reduce(lambda acc, x: acc * x, numbers, 1)    # 120 — explicit start value

The optional third argument seeds the accumulator. Without it the first element is used as the start.

In practice reduce is rarely the clearest choice. Most common reductions have a more expressive built-in:

Instead of reduce(...) Use
Sum sum(iterable)
Product math.prod(iterable)
Maximum / minimum max(iterable), min(iterable)
Any / all any(iterable), all(iterable)
String join "sep".join(iterable)

reduce is the right tool when none of the above fit: composing a list of functions, merging a sequence of dicts, or folding over a custom algebraic structure.

@total_ordering

@total_ordering derives the missing comparison methods automatically. You define __eq__ and one of __lt__, __le__, __gt__, or __ge__, and the decorator fills in the rest:

from functools import total_ordering

@total_ordering
class Version:
    def __init__(self, major, minor):
        self.major = major
        self.minor = minor

    def __eq__(self, other):
        return (self.major, self.minor) == (other.major, other.minor)

    def __lt__(self, other):
        return (self.major, self.minor) < (other.major, other.minor)

v1 = Version(1, 9)
v2 = Version(2, 0)

v1 < v2    # True  — defined directly
v1 > v2    # False — derived by @total_ordering
v1 <= v2   # True  — derived
v1 >= v2   # False — derived

Rust equivalent: #[derive(PartialEq, PartialOrd)] — define equality and one ordering method, derive the rest.

@wraps

When you write a decorator, the wrapper replaces the original function. Without @wraps, tools like help(), logging, and tracebacks show the wrapper’s name and docstring instead of the original’s:

from functools import wraps

def log_calls(func):
    @wraps(func)              # copies __name__, __doc__, __module__, etc.
    def wrapper(*args, **kwargs):
        print(f"calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

@log_calls
def add(a, b):
    """Add two numbers."""
    return a + b

add.__name__   # "add"              — correct, thanks to @wraps
add.__doc__    # "Add two numbers." — preserved

Without @wraps both would show "wrapper" and None. Always apply @wraps when writing decorators.
